    You should keep track of your cross-validated errors and use whatever algorithm produces the most accurate predictions. As a hint, I suggest that you explore some of the individual packages that train() is calling. They might be faster.

    ARMANDO ESCOBARSignature Track· 6 days ago 

    About the NA Values:

    • It’s necessary to read the document “2013.Velloso.QAR-WLE “ on http://groupware.les.inf.puc-rio.br/har and I suggest to take a look to the original data set.

    The research was done for:

    • Weight Lifting Exercises Data Set: is to investigate "how well" this activity was performed by 6 male participants using a dumbbell (1.25kg).

    • Four sensors were used on: forearm, arm, lumbar belt and dumbbell.

    • Data per sensor:

      • Position: roll, pitch, yaw

      • Acceleration - 3 axis (x, y, z)

      • Gyroscope - 3 axis (x, y, z)

      • Magnetometer - 3 axis (x, y, z)

    • Total Data per Sensor: 3 x 4 = 12

    • Project Data Set / Original Data Set: both data sets must be clean because:

      • The researchers included on the data sets same calculations like kurtosis, average, maximal, minimal values, etc.

      • This calculations were made for groups of measurements define by the column “New Window” to filter some noise.

      • This is the cause of having “NA” Values

      • Same calculations made by the researchers, are not necessary wrong, but not in the right column (for example calculate by yourself  max_roll: maximal value for the roll position for a window of measurements)

    • I’m not saying that the conclusions of the researchers are wrong. I only say, that the data set must be clean before using it.

    • Although I did not finish yet the cleaning of the data sets, as initial conclusions:

      • All columns with calculations done by researchers must be deleted

      • Your tidy data set should have the following columns

        • user_name

        • classe

        • raw_timestamp_part_1 / part_2: times for measurements (use POSIXct)

        • new_window - num_window: to see when a group of measurements start / finish.

        • 48 columns with the sensor measurements.

    • It is possible that finally the data set do not need the columns “raw_timestamp_part_1 / part_2” and / or new – num_window, but at least 50 columns defined by the “user_name”, “classe”, and 48 columns with the sensor measurements.

    Finally, there is a reason for deleting “NA” values and some columns.

    Thanks for your insight - I'd reached about the same conclusions.. a number of questions still remains though.

    How to use part_2? What's the unit? I'm guessing it's time elapsed since beginning of window and should be transformed so that measurements are ~0.5 seconds apart, based on the sliding windows discussion in the paper and the fact that this appears to work but I'm not quite sure. This likely won't make much of difference for this course project and could be treated as ordinal, though.

    More importantly, the question on how and whether to treat it as a time-series problem. As a time-series, there are ~850 time series and the outcome is measured only for the whole series. I believe the appropriate way to deal with this is extract the features per time series and then predict with those derived features. For example, http://cran.r-project.org/doc/contrib/Zhao_R_and_data_mining.pdf

    If I understood correctly, this is the approach taken in the original paper, using fairly simple statistical summaries (the other option would be using estimates of time series model parameters as Zhao does). However, as Armando says, both the training set given to us and the original full data set are problematic - the calculated features don't match the data, some are in the wrong column, variables are misnamed, some only have missing values. 

    So, the options for the course projects are:
    a) use the calculated features in the data set, ignoring the mistakes - we don't need the variable names for prediction anyway - we wouldn't have all the derived features listed in the paper but we'd have some or most. The training set only has the derived data for only 406 time series though.
    b) calculate the same derived features as in the paper or more complicated features (this course does not cover such calculations though) from the training data set first for each of the ~850, then split the derived features into training and testing, model - our training data is a random sample from the full data so we're randomly missing measurements from the time series
    c) ignore time and treat measurements as independent observations, even though they are not. You can build a very successful classifier based on this. What this means is unclear to me. Basically, I thought a num_window would be either a set of 10 repetitions or 1 repetition in a set, but if this is the case, we would know from the starting position that the dumbbell was going to be only lifted halfway, which seems odd to me.

    • Is not clear was is the meaning of “raw_timestamp_part_2”
    I had another look and figured it out - it is actually in the article: "sampling rate of 45 Hz" means 45 samples per second, and the unit is millionths of a second.. A new_window begins just before a full second would have passed, and some windows are part of the same set of curls: for example, windows 4-10 are a consecutive 7.5 seconds of Adelmo throwing his hips to the front (class E), windows 11-18 are 7.2 seconds of Carlitos doing a correct curl, and so forth.

    Thus, even the num_windows are not independent observations, but there's a smaller number of time series (~100-200).
